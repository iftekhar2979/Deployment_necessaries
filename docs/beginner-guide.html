Your Workflow, Explained

Let's take apart a real .yml file and see how it works piece by piece.

Part 1: Identity & Trigger

ci-cd.yml
name: CI/CD Pipeline
on:
  push:
    branches: [main]

What it does: This names your automation and tells GitHub: "Only start when someone pushes code to the main branch."

Part 2: The Build Job (Assembly)

ci-cd.yml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run build

What it does: Creates a temporary Ubuntu computer, downloads your code, installs Node.js, and compiles your TypeScript into JavaScript.

Part 3: The Deploy Job (Shipping)

ci-cd.yml
  deploy:
    needs: build # Wait for build to finish!
    steps:
      - uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.EC2_HOST }}
          script: |
            cd /home/ubuntu/app && git pull && pm2 reload all

What it does: Waits for the build job to succeed, then logs into your AWS EC2 server via SSH and tells it to pull the new code and restart the app.

Secrets & Security

Golden Rule: Never, ever hard-code passwords. Use GitHub Secrets.

Dangerous Way

env:
  DB_PASS: "my-secret-123"
# EVERYONE CAN SEE THIS!

Secure Way

env:
  DB_PASS: ${{ secrets.DB_PASS }}
# Hidden & safe in GitHub Settings

Common Beginner Traps

1. Indentation Hell: YAML is based on spaces. One extra space can break the whole pipeline.
2. Missing "Needs": If Job B needs files from Job A, you MUST use the needs: keyword.
3. Old Node Versions: Using Node 14 when your code needs Node 20. Always check your setup-node version!
07

Questions? We have answers.

Q: What if the workflow breaks my app?

A: PM2 keeps the old version running if the new build fails. You can rollback manually in seconds if needed. Automation is safer than manual work!

Q: Can I run this without pushing code?

A: Yes! Go to the Actions tab on GitHub, select your workflow, and click "Run workflow". Perfect for testing.

Q: Is this only for JavaScript/Node.js?

A: No! GitHub Actions works for Python, Java, C++, Go, and every other language. If you can run it on a computer, Actions can automate it.

Q: Can I see what the robots are doing?

A: Yes! Every step has a live log. Click on any running action to see exactly what's happening and where it failed.

Quick Reference Card

Keep these snippets handy while writing your first workflows!

Basic Structure

name: Workflow Name
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest

Common Triggers

# When code is pushed
on: push: branches: [main]

# When PR is created
on: pull_request: branches: [main]

# On schedule (9 AM daily)
on: schedule: - cron: '0 9 * * *'

# Manually trigger
on: workflow_dispatch

Popular Actions

# Download code
- uses: actions/checkout@v4

# Setup Node.js
- uses: actions/setup-node@v4
  with: node-version: 20

# SSH to server
- uses: appleboy/ssh-action@v1.0.3
  with: host: ${{ secrets.HOST }}

# Upload to server
- uses: appleboy/scp-action@v0.1.4

Running Commands

# Simple command
- run: npm install

# Multi-line commands
- run: |
    npm install
    npm run build

# With Env Vars
- run: npm run build
  env: NODE_ENV: production

# With Working Directory
- run: npm build
  working-directory: ./backend

Using Secrets

# Reference in YAML
env: PW: ${{ secrets.PASSWORD }}

// In JS code
const pw = process.env.PW;

# UI: Settings -> Secrets -> Actions

Control Flow

  • if: success() (Default)
  • if: failure()
  • if: always()
  • needs: [job1, job2]

Pro Tips

  • Caching: Use cache: 'npm' in setup-node to save time!
  • Badges: Add status badges to your README to show if builds are passing.
  • Matrix: Use strategy: matrix to test on multiple OS/Node versions at once.
